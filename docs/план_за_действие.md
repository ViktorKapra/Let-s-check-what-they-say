## Фаза 1 — Данни (след одобрение от преподавателя)

### 1.1 Серийно теглене (baseline)
- `requests` + `re` за извличане на `window.__remixContext` от HTML
- Парсиране на JSON — полета: `sessionStatements`, `parlGroups`, `pgMemberships`
- Всяко изказване съдържа: `id`, `paragraphs[]`, `parlMemberId`, `title`, `number`
- Свързване депутат → партия чрез `pgMemberships`
- Запис на резултата като JSON и CSV

### 1.2 Паралелно теглене (asyncio — I/O подход)
- `aiohttp` + `asyncio.gather()` за конкурентни HTTP заявки
- Сравнение с серийното теглене → демонстрира I/O паралелизъм
- Измерване на speedup при различен брой конкурентни заявки

### URL структура на strazha.bg
```
https://www.strazha.bg/sessions/{ДАТА}/steno/{НОМЕР}/
```
Данните са вградени в HTML-а като JSON в `window.__remixContext` — не е нужен JavaScript.

## Фаза 2 — Имплементация (минимум 3 паралелни подхода)

### 2.0 Серийно решение (baseline — задължително)
- Чист Python цикъл
- Word count, n-gram (bigram/trigram), TF-IDF по партия
- Служи като отправна точка за всички сравнения
- **Критерий:** коректност и яснота (10 т.)

### 2.1 CPU локален паралелизъм — `multiprocessing.Pool`
- `Pool.map()` за паралелна обработка на изказвания
- Всяко изказване се обработва независимо → embarrassingly parallel
- Сравнение: `Pool.map()` (блокиращо) vs `Pool.map_async()` (неблокиращо)
- Обяснение: заобикаляне на GIL чрез под-процеси
- **Критерий:** паралелни реализации (25 т.)

```python
from multiprocessing import Pool
import multiprocessing as mp

def analyze_speech(speech):
    # токенизация, word count, n-gram
    ...

with Pool(processes=mp.cpu_count()) as pool:
    results = pool.map(analyze_speech, speeches)
```

### 2.2 Vectorization — NumPy / scipy.sparse
- `scipy.sparse` TF-IDF матрица за целия корпус
- NumPy операции за честотни анализи
- Косинусово сходство между партийни вектори
- Обяснение: SIMD инструкции, оптимизирани C библиотеки
- **Критерий:** data-parallel подход (задължителен по изискванията)

### 2.3 Distributed — Dask или Ray (препоръчителен)
- `dask.bag` или `ray.remote` за разпределен анализ
- Подходящ за обработка на корпус по-голям от RAM
- Разпределяне по народни събрания или по партии
- Обяснение: overhead при комуникация vs. ускорение
- **Критерий:** distributed вариант — допълнителна стойност

### Пайплайн на данните

```
Стенограми (JSON)
    │
    ├── asyncio          → I/O-bound теглене
    │
    ├── multiprocessing  → CPU-bound обработка (токенизация, n-gram)

    ├── NumPy/scipy      → Vectorized TF-IDF матрица
    │
    └── Dask/Ray         → Distributed анализ (по НС / по партия)
```

## Фаза 3 — Измервания (задължително по изискванията)

### Три размера на входа
| Размер | Описание | Прибл. изказвания |
|--------|---------|-------------------|
| Малък | 1 заседание | ~280 |
| Среден | 1 месец (~8–10 заседания) | ~2 500 |
| Голям | 1 година (~120 заседания) | ~35 000 |

### Изисквания към измерванията
- Поне **3 повторения** на всеки експеримент + осредняване
- Инструмент: `time.perf_counter()`
- Метрики: **времe за изпълнение**, **speedup**, **efficiency** (при възможност)
- Фиксиране на случайността там, където е приложимо

### Формула за speedup и efficiency
```
Speedup(p)   = T_serial / T_parallel(p)
Efficiency(p) = Speedup(p) / p
```

### Изисквана документация на средата
- CPU: модел, брой ядра/нишки
- RAM: обем
- GPU: ако се използва
- Python версия
- Версии на всички библиотеки (`requirements.txt`)

### Задължителни артефакти
- Поне **1 таблица** с резултатите
- Поне **1 графика** (speedup vs. брой процеси)

## Бележки

- Данните на strazha.bg са публична информация по ЗДОИ
- strazha.bg финансира се от дарения — при масово теглене е добра практика да се свържете с екипа
- Препоръчва се `asyncio` да се изпълнява с `asyncio.Semaphore(10)` за ограничаване на конкурентните заявки
- Серийното решение трябва да е написано преди паралелните — много по-лесно е да се паралелизира работещ код
